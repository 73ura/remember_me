# PRレビュー規約

## 1. レビューの基本姿勢

* **敬意と建設的フィードバック**

  * 個人ではなくコードにフォーカスする。
  * 改善提案は「なぜ」や「どう良くなるか」を添える。
* **最小限でマージ可能に**

  * ブロッカー（重大なバグ・セキュリティリスク）があれば明確に指摘。
  * 細かい改善点は「Optional」「Nice to have」と明示。
* **一貫性を優先**

  * プロジェクトのスタイルガイドや既存実装に合わせる。
  * 「独自の好み」ではなく「チームの基準」を優先する。

---

## 2. レビュー観点

### A. コード品質

* フロントエンド(Vue.js)

  * ESLint / Prettier が通っているか。
  * 冗長な記述や再利用可能なコンポーネント化の不足がないか。
  * Vueのリアクティブデータの扱い（`ref`, `reactive`）が正しく行われているか。
* バックエンド(Rails)

  * Fat Controllerを避け、Model/Service層に責務が整理されているか。
  * バリデーション・認可処理が適切に行われているか。
  * N+1クエリや不要なSQL発行がないか。
* DB(PostgreSQL)

  * マイグレーションは冪等で安全に実行/rollbackできるか。
  * インデックスや制約（NOT NULL, FK, unique）が正しく設計されているか。

### B. テスト

* フロント: Vitest

  * 単体テストが主要ロジックをカバーしているか。
  * DOM操作やイベントのテストが意図通りか。
* バック: RSpec + FactoryBot

  * モデルのバリデーション・関連・スコープを網羅しているか。
  * Request specで成功/失敗ケースを両方検証しているか。
  * Devise test helpers / OmniAuth mock を活用して認証関連をテストしているか。
* CI (GitHub Actions)

  * ESLint / Prettier / Vitest / RSpec がすべて成功しているか。

### C. セキュリティ & パフォーマンス

* 認可チェック（権限確認）が漏れていないか。
* ユーザー入力に対するサニタイズ / ストロングパラメータが適切か。
* APIレスポンスで不要な情報（パスワードハッシュ等）が漏れていないか。
* 不要な再レンダリングやクエリ負荷が発生していないか。

### D. 可読性 & 保守性

* 関数/メソッド/クラス名は意図が明確か。
* コメントや型注釈が必要十分か。
* 同じ処理を繰り返していないか（DRY原則）。

### E. ユーザー体験 (UI/UX)

* UIの変更がある場合はスクリーンショットや動画で確認できるか。
* i18n対応（日本語/英語の文言管理）が適切か。
* アクセシビリティ（ラベル、フォーカス制御）が考慮されているか。

## 3. レビューフロー

1. **PRの概要確認**

   * PRテンプレートに従い、目的・変更内容・動作確認手順が明記されているか。
2. **CI結果の確認**

   * すべてのチェック（Lint / Test）が成功しているか。
3. **コードレビュー**

   * 主要変更ファイルを中心に観点ごとに確認。
   * 不要な差分（console.log, binding.pry, コメントアウト残し）がないか。
4. **動作確認**

   * 手順に従ってローカルまたはDocker環境で再現できるか。
5. **レビューコメント**

   * 指摘レベルを明示:

     * **Must**: マージ前に修正必須
     * **Should**: 望ましいが必須ではない
     * **Nice to have**: 今回はマージ可能、将来改善
6. **承認/再レビュー**

   * 基準を満たせば Approve。
   * 改善必要なら Request changes。
   * 軽微な改善提案なら Comment。

---

## 4. 禁止事項

* 個人攻撃や感情的な言葉を使うこと。
* 根拠のない「なんとなく変」コメント。
* 「LGTM（Looks Good To Me）」だけでApproveすること。

---

## 5. 最終責任

* レビュアーは「品質の担保」に責任を持つ。
* マージ担当は「影響範囲・リリース判断」に責任を持つ。
